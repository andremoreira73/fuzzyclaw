## Critical Findings

### 1. Cross-user API access is not scoped

Severity: Critical

Files:

- [core/api_views.py](/home/memology/Documents/git_folders/fuzzyclaw/core/api_views.py#L59)
- [core/api_views.py](/home/memology/Documents/git_folders/fuzzyclaw/core/api_views.py#L71)
- [core/api_views.py](/home/memology/Documents/git_folders/fuzzyclaw/core/api_views.py#L86)
- [fuzzyclaw/settings/base.py](/home/memology/Documents/git_folders/fuzzyclaw/fuzzyclaw/settings/base.py#L96)
- [core/tests.py](/home/memology/Documents/git_folders/fuzzyclaw/core/tests.py#L540)

What is wrong:

- The DRF default permission is only `IsAuthenticated`.
- `BriefingViewSet`, `RunViewSet`, and `AgentRunViewSet` expose global querysets with no `get_queryset()` scoping and no object-level permission checks.
- Any authenticated user can retrieve, update, or delete another user's briefings/runs/agent-runs if they know or can enumerate IDs.

Why it matters:

- This is a direct multi-tenant data isolation break.
- It also lets one user create or manipulate execution records tied to another user's data.

Recommended fix:

- Scope every queryset to `request.user`.
- Add object-level permission checks.
- Add regression tests for cross-user list/detail/update/delete access.

### 2. Markdown rendering is a stored XSS sink

Severity: Critical

Files:

- [core/templatetags/markdown_extras.py](/home/memology/Documents/git_folders/fuzzyclaw/core/templatetags/markdown_extras.py#L19)
- [core/templatetags/markdown_extras.py](/home/memology/Documents/git_folders/fuzzyclaw/core/templatetags/markdown_extras.py#L30)
- [templates/core/run_detail.html](/home/memology/Documents/git_folders/fuzzyclaw/templates/core/run_detail.html#L57)
- [templates/core/run_detail.html](/home/memology/Documents/git_folders/fuzzyclaw/templates/core/run_detail.html#L90)

What is wrong:

- Markdown is converted to HTML and immediately marked safe with `mark_safe(...)`.
- The rendered HTML is inserted into coordinator and agent report views.
- Reports are generated by LLMs and may contain scraped or adversarial content.

Why it matters:

- A malicious report can persist arbitrary HTML/JS and execute in an authenticated operator's browser.
- This is especially risky because the system explicitly ingests untrusted external content.

Recommended fix:

- Sanitize rendered HTML before marking it safe, or render markdown with raw HTML disabled.
- Add an XSS regression test that submits markdown containing `<script>` or dangerous attributes.

### 3. Container slot accounting leaks on startup failures

Severity: Critical

Files:

- [core/containers.py](/home/memology/Documents/git_folders/fuzzyclaw/core/containers.py#L384)
- [core/containers.py](/home/memology/Documents/git_folders/fuzzyclaw/core/containers.py#L500)
- [core/agent_tools.py](/home/memology/Documents/git_folders/fuzzyclaw/core/agent_tools.py#L137)
- [core/agent_tools.py](/home/memology/Documents/git_folders/fuzzyclaw/core/agent_tools.py#L155)
- [core/tests.py](/home/memology/Documents/git_folders/fuzzyclaw/core/tests.py#L814)

What is wrong:

- `_container_count` is incremented before Docker client creation, volume validation, env preparation, and `client.containers.run(...)`.
- If any of those later steps fail, the slot is never released.
- The caller deletes the `AgentRun`, but does not decrement the counter.

Why it matters:

- Repeated launch failures can permanently exhaust `FUZZYCLAW_MAX_CONTAINERS` until cleanup or worker restart.
- The platform can start returning false concurrency-limit failures even when no containers are actually running.

Recommended fix:

- Wrap post-increment setup in `try/except` and always roll back the counter on failure.
- Add a test that forces `client.containers.run(...)` or volume validation to fail and verifies rollback.

### 5. Volume allow/block policy is vulnerable to symlink bypass

Severity: Critical

Files:

- [core/containers.py](/home/memology/Documents/git_folders/fuzzyclaw/core/containers.py#L92)
- [core/containers.py](/home/memology/Documents/git_folders/fuzzyclaw/core/containers.py#L104)
- [core/containers.py](/home/memology/Documents/git_folders/fuzzyclaw/core/containers.py#L473)

What is wrong:

- Volume validation is based on normalized path strings, not canonical filesystem paths.
- The code never resolves symlinks before applying allowlist/blocklist checks.

Why it matters:

- A path under an allowed prefix can point via symlink to a blocked location such as `/etc`.
- That defeats the intended host-path security boundary.

Notes:

- This is an inference from the implementation; I did not execute an exploit.

Recommended fix:

- Resolve host paths with `Path(...).resolve()` or `os.path.realpath()` before validation.
- Add tests covering symlink traversal into blocked paths.

### 6. Early agent startup failures do not emit structured failure artifacts

Severity: Critical

Files:

- [agent_runner.py](/home/memology/Documents/git_folders/fuzzyclaw/agent_runner.py#L140)
- [agent_runner.py](/home/memology/Documents/git_folders/fuzzyclaw/agent_runner.py#L145)
- [agent_runner.py](/home/memology/Documents/git_folders/fuzzyclaw/agent_runner.py#L228)
- [core/containers.py](/home/memology/Documents/git_folders/fuzzyclaw/core/containers.py#L514)
- [core/agent_tools.py](/home/memology/Documents/git_folders/fuzzyclaw/core/agent_tools.py#L227)

What is wrong:

- Parsing the agent file, constructing the model, and building tools all happen before the `try` block that writes `error.json` and signals Redis.
- If any of those early steps fails, the process exits without producing the structured failure files the coordinator expects.

Why it matters:

- Failures degrade into timeout/crash detection instead of immediate, actionable errors.
- Operators lose diagnostic fidelity exactly when startup is broken.

Recommended fix:

- Move all startup work inside the protected `try` block.
- Add tests for parse/model/tool-construction failures and assert `error.json` is written.

### 7. The web service has direct host Docker control

Severity: Critical

Files:

- [docker-compose.yml](/home/memology/Documents/git_folders/fuzzyclaw/docker-compose.yml#L35)
- [docker-compose.yml](/home/memology/Documents/git_folders/fuzzyclaw/docker-compose.yml#L37)
- [docker-compose.yml](/home/memology/Documents/git_folders/fuzzyclaw/docker-compose.yml#L40)

What is wrong:

- The `web` container mounts `/var/run/docker.sock`.
- The service is also exposed on port `8200`.

Why it matters:

- Any Django-side RCE or template injection becomes host-level Docker control.
- This is a large blast-radius increase for the internet-facing surface.

Notes:

- This may be an intentional architecture choice, but it is still a critical operational risk that should be documented and isolated.

Recommended fix:

- Remove Docker socket access from `web` if it is not strictly required there.
- Restrict Docker control to the worker that actually needs it, or isolate it behind a narrower control surface.

Severity: Warning

What is wrong:

- `FUZZYCLAW_HITL_TIMEOUT` defaults to 1800 seconds.
- `FUZZYCLAW_AGENT_TIMEOUT` defaults to 600 seconds.

- Human-in-the-loop flows can fail by design under normal use.
- The system will kill the waiting agent before the advertised timeout expires.

Recommended fix:

- Enforce `HITL_TIMEOUT <= AGENT_TIMEOUT`, or extend the runtime when a tool explicitly enters HITL wait mode.

## Overall Assessment

The codebase is in decent shape mechanically and the existing Django test suite provides good baseline safety, but there are several high-severity trust-boundary issues:

- API authorization is not multi-user safe.
- Markdown rendering trusts untrusted content.
- Container/runtime orchestration has multiple failure-path gaps.
- Deployment/orchestration defaults are easy to run in an unsafe mode.

I would not treat the current state as production-ready for multiple users until the critical items are addressed.
